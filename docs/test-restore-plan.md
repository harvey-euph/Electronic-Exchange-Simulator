# Test Correctness & Restore Script Architecture

## 1. Goal
Instead of a completely separate script, we will directly extend the existing `test-correctness` script framework. The script will be refactored to support testing uninterrupted execution (Control) and execution interrupted by crashes (Experimental) in one unified flow. 

For any given test directory (e.g., `tests/cross-multiple`), the script will create 3 sub-directories:
1. **`expected/`**: Generated by the offline Python simulator (`scripts/generate-expected`).
2. **`control/`**: Generated by running the C++ services continuously over `requests.csv` followed by `requests-conti.csv` (if it exists).
3. **`experimental/`**: Generated by running the C++ services over `requests.csv`, killing the services, restarting them (triggering recovery), and then running `requests-conti.csv`.

Ultimately, the test passes if `expected/` == `control/` == `experimental/`.

## 2. State to Dump & Compare
Each of the 3 directories (`expected`, `control`, `experimental`) will contain identical file structures:
1. **`executions.csv`**: The stream of order executions.
2. **`book-core.csv`**: Dump of the central limit order book directly from `MatchingEngine`.
3. **`book-server.csv`**: Dump of the Level 3 market data reconstructed by `MarketDataServer`.
4. **`book-client.csv`**: Dump from the downstream client application (`book-client-dumper`).
5. **`positions.csv`**: Dump of all user balances/positions from the `ClientDatabase`.
6. **`open-orders.csv`**: Dump of all active open orders managed by the `ClientDatabase`.

## 3. Implementation Steps

### Step 3.1: Upgrade Services with Dump Hooks
- **ME Orderbook Dump**: Add `ME_DUMP_FILE` support to `matching-engine.cpp` to dump its state to `book-core.csv`.
- **DB State Dump**: Add a hook in `client-manager.cpp` (e.g., via SIGINT or an environment variable `DB_DUMP_DIR`) to dump `positions.csv` and `open-orders.csv`.
- **Python Simulator Update**: Update `scripts/generate-expected` so it also produces `book-core.csv`, `positions.csv`, and `open-orders.csv` into the `expected/` directory.

### Step 3.2: Extend `test-correctness` (or rename to `test-pipeline`)
1. Create directories: `expected/`, `control/`, `experimental/`.
2. **Expected Phase**: Run `generate-expected` to populate `expected/`.
3. **Control Phase**:
   - Clean environment (shm, logs).
   - Start ME, MD, Client Manager.
   - Inject `requests.csv` and `requests-conti.csv`.
   - Send SIGINT to gracefully dump all files into the `control/` folder.
4. **Experimental Phase**:
   - Clean environment.
   - Start ME, MD, Client Manager.
   - Inject `requests.csv`.
   - **Hard Kill** (-9) the ME, MD, and DB services.
   - **Restart** ME, MD, DB.
   - Inject `requests-conti.csv`.
   - Send SIGINT to gracefully dump all files into the `experimental/` folder.
5. **Verification**:
   - Recursively `diff -q expected/ control/`
   - Recursively `diff -q expected/ experimental/`
